The Most Skipped Step

Think before you prompt

01Does this already exist? Check Notion, Airtable, Linear — before you generate a single line.
02Who is the actual user of this app?
03What is the one thing they need to do?
04What is the core data model — the entities?
05What should NOT be in v1?
The platform will generate something no matter what you give it. The question is whether it's useful.
What Not To Do

What a bad prompt looks like

"Make me an app where people can log their workouts and track progress over time. Should have a nice dashboard with charts and stats. Would be cool if friends can see each other's progress too. Maybe some kind of streak or badge system. Oh and it should work on mobile."

No user defined

"people" — what people? That changes everything about the UI.

Scope creep baked in

"friends", "badges", "streaks" — 3 features added with the word "maybe".

No data model

What is a workout? The AI will guess — and you'll spend hours fixing it.

"Nice" is not a spec

The AI has no reference for your taste. You will hate what it builds.

What To Do Instead

What a good prompt looks like

"Build a workout logger for intermediate weightlifters who already track sets, reps, and weight on paper. Data model: a Workout has a date and many Exercises. An Exercise has a name, and many Sets with reps and weight (kg). One screen to add a workout, one to view history as a list. No social features, no badges, no charts. Web only, desktop-first."

Specific user

Intermediate weightlifters — the AI now knows the UI feel.

Scope bounded

"No social features, no badges" — explicit non-goals stop scope creep.

Data model defined

Workout → Exercises → Sets with reps and weight in kg. No guessing.

Platform decided

"Web only, desktop-first" — one platform, one layout. No wasted work.

The Skill

The prompt is the skill

Vague
"Build me a habit tracker app with a nice UI and some stats"
Structured
"Build a React + Vite habit tracking app for busy professionals. Core entities: Habit (name, frequency, streak), DailyLog (date, completed). Screens: dashboard with today's habits, habit detail with 30-day history. Mobile-first. No auth in v1."
The prompt is the skill. Not the generation — the prompt.

Iteration 1 of 3

Fix a missing feature

Prompt
"Add a 30-day history chart for each habit showing completion rate as a bar chart. Use the existing data model."
What to narrate
Does it use the existing data model or invent a new one?
Did it pick an appropriate chart library?
Did it break anything that was already working?
Does the new feature feel integrated or bolted on?
Iteration 2 of 3

Fix a UX issue

Prompt
"The layout breaks on mobile. Fix spacing on the dashboard so items stack correctly below 768px. Keep the desktop layout unchanged."
What to narrate
Does it understand responsive design or just apply class names?
Did it test on both breakpoints or just fix what it could see?
Did the fix introduce regressions elsewhere?
Would you approve this in a PR?
Iteration 3 of 3

Into the code

Architecture

Are components sensibly separated? Is there a clear data layer?

Naming

Do variables and functions mean what they say? Or did it use 'data', 'item', 'thing'?

Error handling

What happens when something goes wrong? Does it fail silently?

State management

Is complexity growing faster than the structure that contains it?

Find the one thing a prompt can't fix. The moment you need to change it — you're a developer again. That's the point.

The Hard Truths

Where it struggles

Complex business logic — it approximates, doesn't reason
Multi-tenant authentication — gets the shape wrong
Payments and compliance — regulated industries, real liability
State management at scale — structure collapses under complexity
Accessibility — skipped unless explicitly prompted, still often wrong
Context drift — long sessions forget earlier decisions
Real-world messy data — built to impress in demos, not handle edge cases
Security — Guardio Labs (April 2025) found prompt injection vulnerabilities in AI-generated code
Concrete Evidence

The real-world data problem

Same prompt. Same app. Three platforms. All three handled their own generated sample data fine. Then a real CSV was uploaded.

Bolt.new

Database error on first real post. Defaulted to "paste a URL" instead of file upload.

Lovable

Could not parse real CSV on first attempt. Required follow-up prompts to fix.

Replit

Dumped raw API response to screen. Needed one more prompt — but it was fixable.

These tools are trained to impress you in demos. Real-world messy data is where you find the seams.
